🧐 Skeptical Assessment

Skeptical Review: OKF Schema Skill Evaluation — iteration-30.06.26-13.12

Executive Summary

Verdict: The skill demonstrates ZERO measurable advantage in this iteration.

Both conditions scored identically across all three evals (22/29 assertions, 75.9% pass rate). The delta is flat. This is either evidence that the skill provides no value, or that the evaluation methodology is flawed. Below is a ruthless critique.

---

Methodology Flaws

1. Assertions Check the Wrong Scope

Several assertions use `output_matches` regex against the entire transcript, not just the final validation output:

  • E1/E2/E3/W2/W5 checks in Eval 1: The regex `^(?!.\[E1\]).$` scans the full transcript. Since the transcript includes the initial validation run (which intentionally reports these errors before fixing them), the assertion fails even though the final output is clean. This is a grader bug, not an agent failure.
  • Cross-link check in Eval 2: The regex `\[.+?\]\(.+?\.md\)` looks for markdown link syntax in the transcript text, not in the actual concept files. The agent described cross-links in prose ("lithium-ion → thermal/cooling-systems.md") rather than embedding raw markdown links in the transcript. The files themselves contain proper cross-links.
Impact: 5 false negatives in Eval 1, 1 false negative in Eval 2. These are grading artifacts, not skill gaps.

2. The Tasks Are Too Easy

All three evals involve basic OKF operations that any agent with CLI help and the spec can figure out:

  • Adding missing frontmatter fields
  • Fixing date formats
  • Running `lint`, `index`, `validate --strict`

The OKF spec is well-written and self-contained. The CLI commands are intuitive. There is no deep reasoning or edge-case handling required. A skill cannot demonstrate value when the baseline is already sufficient.

3. No Negative-Space Testing

The evals don't test scenarios where the skill would genuinely help:

  • Custom schema authoring: Writing JSONSchema from scratch
  • Complex migration: Handling wikilinks, Obsidian vaults, or CSV imports
  • Error interpretation: Understanding nuanced validator output and choosing the right fix
  • Workflow sequencing: Knowing why index → lint → validate is the right order

The skill's "Tips for Agents" section contains valuable workflow guidance (the three-command pipeline, check mode, comment preservation). None of this is tested.

---

Results Interpretation

Pass/Fail Rates Reflect Grader Limitations, Not Knowledge

The identical scores are driven by:

  • Shared failures: Both conditions hit the same grader bugs (E1/E2/E3 pattern checks, cross-link transcript check).
  • Shared successes: Both conditions correctly ran the CLI commands and achieved conformant bundles.

The pass/fail rates do not reflect whether the with-skill agent had deeper understanding or made fewer mistakes. They reflect whether the transcript text matched regex patterns.

Where the Skill Provides No Advantage

  • Basic CLI usage: `validate --strict`, `lint`, `index` are discoverable from `--help`.
  • Frontmatter syntax: The OKF spec §4.1 is explicit and short.
  • Simple fixes: Adding `type:`, `timestamp:`, fixing ISO dates — trivial once the error is reported.

Where the Skill Would Shine (But Is Untested)

  • Schema-aware validation: Understanding that `_schema/` files must match the `type` field case-exactly (e.g., `Concept.schema.json` for `type: Concept`). The without-skill agent would likely stumble here.
  • Round-trip YAML preservation: Knowing that `ruamel.yaml` preserves comments and quotes.
  • The index → lint → validate workflow: The skill explicitly recommends this sequence. Without it, an agent might validate before linting, wasting cycles.
  • Custom schema development: Writing JSONSchema definitions is non-trivial and the skill provides examples.

---

Bias Detection

Assertion Bias

  • Eval 1: The E1/E2/E3/W2/W5 `output_matches` assertions are biased toward a transcript that only contains the final validation output. Any agent that logs their process (as instructed) will fail these. This penalizes transparency.
  • Eval 2: The `okf_version` check expects frontmatter in the root `index.md`. But `okf-schema index` regenerates this file without frontmatter. The init-generated `index.md` (with `okf_version`) gets overwritten. This tests implementation detail, not correctness.

Grading Strictness

The grader is appropriately strict for `command_executed` and `file_exists` checks, but the `output_matches` regex checks are brittle and prone to false negatives.

---

Actionable Recommendations

Immediate Fixes

  • Fix Eval 1 assertions: Change E1/E2/E3/W2/W5 checks from `output_matches` on the full transcript to `output_contains` on the final validation output only. Or add a "final validation output" section marker and scope checks to that region.
  • Fix Eval 2 cross-link check: Check the actual concept files for markdown links (`file_glob_contains` with regex), not the transcript.
  • Fix Eval 2 okf_version check: Either expect `okf_version` in a separate metadata file, or modify `okf-schema index` to preserve frontmatter, or remove this assertion.

New Evals to Add

  • custom-schema-authoring: Ask the agent to write a JSONSchema for a new concept type and validate it. Tests skill-specific knowledge about schema discovery.
  • obsidian-migration: Provide an Obsidian vault with wikilinks, tags, and frontmatter. Test whether the agent knows the conversion workflow from the skill's references.
  • error-interpretation: Provide a bundle with subtle errors (e.g., nested lists inside tags, schema type mismatch). Test whether the agent correctly diagnoses and fixes without running validate first.
  • workflow-sequence: Give a broken bundle and explicitly ask the agent to follow "the recommended workflow." The with-skill agent should run index → lint → validate; the without-skill agent might run validate first.

Remove or Modify

  • Eval 1 E1/E2/E3/W2/W5 pattern checks: Replace with scoped checks or remove. They measure transcript formatting, not correctness.
  • Eval 2 okf_version: Remove or redefine. It tests an implementation quirk, not OKF conformance.

---

Conclusion

This iteration does not demonstrate skill value, but the reason is methodological, not substantive. The evals are too easy, the grader has false-negative bugs, and the skill's unique knowledge (workflow tips, schema conventions, round-trip preservation) is never exercised.

Recommendation: Fix the grader bugs, add harder evals that test skill-specific knowledge, and re-run. A well-designed eval should show the skill providing a 20-40% improvement on non-trivial tasks.
